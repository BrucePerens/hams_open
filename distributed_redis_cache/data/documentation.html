Distributed Redis Cache

The Distributed Redis Cache module provides a high-performance, horizontally scalable caching solution for Odoo cluster environments. It ensures that all Odoo servers in your cluster stay synchronized by sharing cache invalidation signals through Redis.

Overview

In a multi-server setup, standard Odoo servers keep their own local memory caches. When data is updated on one server, others might still serve old data. This module solves this "cache drift" by providing a central Redis-backed signaling system.

Core Functionality

  1. Instant Synchronization: Whenever a record is updated, the module sends a notification through the database.
  2. Automated Distribution: A background service (the Cache Manager) picks up these notifications and publishes them to Redis.
  3. Cluster-Wide Invalidation: All Odoo servers listening to Redis receive the signal and immediately clear the relevant parts of their local memory.
Smart Invalidation: Unlike standard Odoo, which may clear large portions of the cache at once, this module uses fine-grained invalidation to only flush specific models, keeping the rest of your system fast.

User Instructions

Managing the Cache

Administrators can access the management interface via the Distributed Cache menu in the Odoo navbar.

Checking System Health

Use the Check Redis Status button to verify that Odoo can communicate with the Redis server.

  • A green success message indicates a healthy connection.
  • A red error message means Redis is unreachable. In this case, the module automatically switches to "Fail-Open" mode, falling back to standard local caching to ensure your site stays online.

Manual Cache Clearing

If you suspect that data is out of sync across your cluster, you can manually trigger a flush for a specific data type (a "Model"):

  1. Select the Model (e.g., User or Partner) from the dropdown list.
  2. Click Invalidate Model Cache.
  3. Every server in the cluster will immediately clear its local cache for that specific model.

Multi-Tenant Awareness

This module is fully multi-tenant aware. Cache entries are isolated per website based on the website_id and per company based on the company_id in the request context. This ensures that personal or group-specific data is correctly cached and never leaked between different sites or companies hosted on the same Odoo instance.

Technical Benefits

  • Zero-Sudo Architecture: Built for security, the module uses dedicated service accounts and minimal privileges.
  • Resilience: Designed to be "Fail-Open," meaning your system will never crash due to a Redis outage; it will simply revert to standard Odoo behavior.
  • Efficiency: Uses the SCAN command for Redis operations to ensure performance even with millions of cache keys.
Developer Tip: Use the @distributed_cache() decorator on your custom model methods to automatically benefit from cluster-wide caching and multi-tenant isolation. Ensure that your method arguments are JSON-serializable for optimal performance. The system handles Odoo recordsets, dates, and basic Python collections automatically.

Troubleshooting

If you notice the cache is not invalidating as expected:

  • Check the Cache Manager Daemon: Ensure the background service is running. It is responsible for bridging database events to Redis.
  • Redis Connectivity: Use the "Check Redis Status" button. If Redis is down, the system will fall back to local memory, which may cause temporary drift between servers.
  • Wait for Invalidation: Invalidation signals are processed asynchronously. They usually take less than a second to propagate across the cluster.
