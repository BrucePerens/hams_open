Database Management & SRE Toolkit Guide

The Database Management module is a comprehensive suite designed for Site Reliability Engineers (SREs) and Database Administrators (DBAs). It allows for deep monitoring, performance tuning, and high-availability orchestration of the PostgreSQL database directly from within Odoo.

Getting Started

All database management features are located under the Database & SRE main menu. Access is strictly limited to users with System Administrator privileges to ensure the security and integrity of your data.

1. Monitoring Table Health & Bloat

Database "bloat" occurs when many records are updated or deleted, leaving behind "dead tuples" that still occupy disk space. Over time, this can significantly slow down your system.

How to identify bloat:

  1. Go to Database Health > Table Health & Bloat.
  2. Look at the Dead % column. A value over 20% generally indicates that the table could benefit from a manual cleanup.
  3. The Cache Hit % indicates how much data is being read from memory vs. disk. Higher is better (ideally > 95%).

How to reclaim space:

  1. Select the tables you wish to clean in the list view.
  2. Click Vacuum Analyze Selected. This triggers the PostgreSQL VACUUM ANALYZE command, which removes dead tuples and updates optimizer statistics.
    Performance Note: The system automatically batches these requests to minimize overhead.

Index Advisor:

Tables that are large and have many sequential scans are prime candidates for new indexes.

  • Navigate to Database Health > Index Advisor.
  • The advisor lists tables with high sequential scan counts. Review these and consider if adding an index on columns used in WHERE clauses or JOINs would improve performance.

Security Note: Vacuuming is an intensive operation. Only users with Database Manager privileges can trigger it.

Proactive Alerting: The system automatically monitors these levels and will trigger a PagerDuty incident if critical tables exceed safe bloat thresholds.

2. Index Health & Usage

Indexes speed up read operations but slow down writes and consume disk space. Unused indexes should be removed.

  • Navigate to Database Health > Index Health.
  • The Total Scans column shows how often an index is actually used. If a large index has zero or very few scans, it may be a candidate for removal.

3. Slow Query Tracking (APM)

Application Performance Monitoring (APM) is built-in to help you find the "heaviest" queries hitting your database.

  • Go to Database Health > Slow Queries (APM).
  • Queries are sorted by Total Time. This helps you identify which parts of the application are causing the most load, even if individual executions are fast.
  • Reset Statistics: If you have resolved performance issues and want to start fresh, use the Reset Statistics button in the header.
  • Explain Plan: Click the Explain (search icon) next to any query to generate a detailed EXPLAIN (ANALYZE, BUFFERS) plan. This reveals exactly how PostgreSQL is executing the query and where the bottlenecks are.

4. Active Sessions & Kill Switch

Sometimes a single "runaway" query or a deadlocked session can freeze parts of the system.

Replication Monitoring:

For systems with standby nodes, monitoring replication lag is critical to ensure data consistency and high availability.

  • Navigate to Database Health > Replication Monitoring.
  • The view shows connected standby nodes, their state, and Replay Lag.
  • Go to Database Health > Active Sessions.
  • Review the Duration and the Active Query text.
  • If you identify a process that is causing issues, select it and click Kill Selected Queries. This safely terminates the database backend for that specific session.

Security Note: Terminating sessions can interrupt user operations. Only Database Managers are authorized to use this feature.

5. Performance Optimization Wizard

Configuring PostgreSQL for Odoo can be complex. Our wizard automates this using industry-standard DBA formulas.

  1. Go to Configuration & Tuning > Optimization Wizard.
  2. Enter your server's Total RAM and CPU Cores.
  3. Select your storage type (SSD is highly recommended).
  4. Click Calculate & Apply Optimizations.

Note: Some settings (like shared_buffers) require a full PostgreSQL service restart to take effect. The wizard will notify you if a restart is needed.

6. High Availability (HA) Orchestrator

For production environments requiring high availability, this tool generates configuration templates for Patroni, etcd, and PgBouncer.

  1. Go to Configuration & Tuning > High Availability Orchestrator.
  2. Enter the IP addresses for your Primary and Secondary nodes.
  3. Set a strong Replication Password (minimum 8 characters).
  4. Click Generate Configuration to receive tailored YAML and INI files for your cluster nodes.

Security & Architecture

To maintain a "Zero-Sudo" environment, this module uses a dedicated service account (user_database_management_service) for all sensitive operations. This ensures that even administrative actions are performed with the minimum necessary privileges and are fully auditable.

Multi-Tenant Considerations

Models in this module are classified as logically global. This is because they interface directly with PostgreSQL system views that track the health and activity of the entire database instance. In a multi-company setup, these metrics provide a unified view of the shared database resources, ensuring that administrators can optimize performance for all users simultaneously.
