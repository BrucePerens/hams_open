Binary Downloader Facility

The Binary Downloader is a secure system that provides technical dependencies (helper programs) like kopia (for backups), etcd (for distributed configuration), or cloudflared (for secure tunnels) to Odoo automatically.

1. Purpose and Automation

Modern Odoo deployments often rely on external high-performance tools written in languages like Go or Rust. Instead of requiring manual system administration to install these tools on the host server, the Binary Downloader automates the lifecycle of these dependencies. This module operates without administrative privileges (Zero-Sudo) and ensures that all binaries are verified for integrity before execution.

When an Odoo module needs a specific program, it asks the Binary Downloader to "ensure" it is available. If the program is missing or invalid, it is downloaded, verified for integrity using cryptographic hashes, and installed into a protected internal directory.

2. Operating the Module

While most operations happen automatically when other Odoo modules require them, administrators can monitor, configure, and trigger installations manually. This module supports both system-wide "Manifests" (simple, automatic) and fine-grained "Tenant Versions" (advanced control).

2.1. Monitoring and Manual Installation

  1. Navigate to Settings -> Technical -> Binary Manifests.
  2. Review the list of binaries. The Installed column indicates if the dependency is currently available on the system PATH or in Odoo's private binary directory (hams_bin).
  3. To manually install or update a dependency, click the Install button on the list view, or open the record and click Install Now.
  4. If an installation fails, a detailed notification will explain the cause (e.g., network error, checksum mismatch, or missing member in archive).

2.2. Versioned Binaries and Tenant Assignments

In advanced multi-tenant environments, you may need different websites to use different versions of the same tool.

  1. Version Management: Under Settings -> Technical -> Binary Versions, you can define multiple specific releases for a Software Manifest (e.g., Kopia 0.12.1, 0.13.0).
  2. Tenant Links: Under Settings -> Technical -> Tenant Executables, you can link a specific Website to a Software and select an Active Version.
  3. Automatic Symlinks: The system automatically creates a secure "shortcut" (symlink) in a website-specific directory (tenant_bins/site_ID/). This allows your website-specific scripts to always find the correct version of the tool.

2.3. Adding New Binaries

If you need to add a new tool for a custom module:

  1. Click the New button in the Binary Manifests list.
  2. Binary Name: Enter the command name (e.g., my-utility). Do not use slashes or backslashes.
  3. Company: Select the company that owns this manifest. In multi-company environments, this ensures that each company can have its own specific version or download source for a binary.
  4. Download URL: Provide the HTTPS link to the raw binary or .tar.gz archive.
  5. SHA-256 Checksum: Provide the expected SHA-256 hash of the file. This is mandatory for security.
  6. Archive Type: Select "Raw Binary" if the URL points directly to the executable, "Tarball (.tar.gz)" for Gzipped Tar archives, or "Zip Archive (.zip)" for ZIP files.
  7. Extract Member: For tarballs and zip files, specify the filename within the archive that should be extracted as the executable. This is required for .tar.gz and .zip files.

3. Security Architecture

Security is at the core of the Binary Downloader design:

  • Cryptographic Verification: Every downloaded file is verified against a SHA-256 checksum. If the file's digital fingerprint doesn't match the expected value, the installation is aborted to protect against tampering.
  • Advisory Locking: The system uses database-level locks (PostgreSQL advisory locks) to ensure that multiple Odoo workers don't try to install or update the same binary simultaneously, preventing file corruption.
  • Micro-Privilege: Downloads and installations are performed using a restricted service account (user_binary_downloader_service) to minimize system risk, adhering to the Zero-Sudo mandate.
  • Path Sanitization: Strict protections are in place to prevent "Tar Slip" and "Zip Slip" vulnerabilities. Symbolic links and hard links within archives are strictly forbidden.
  • Isolation: Binaries are stored in a dedicated hams_bin folder within Odoo's data directory, kept separate from system-wide binaries and set with restricted 0o750 permissions.

4. Troubleshooting Common Issues

  • Checksum Failures: If you see a "Security Alert: Checksum mismatch", it means the file downloaded from the internet does not match the fingerprint in Odoo's database. This could be due to a corrupted download or a change in the remote file. Contact your technical support.
  • System Architecture: Automatic installation is currently optimized for 64-bit Linux (x86_64) systems. Other architectures may require manual installation via the system package manager.
  • Network Connectivity: The Odoo server must have HTTPS access to the download URLs (e.g., github.com) to fetch the dependencies. The system also supports local mirrors if configured.
  • Missing Member: For archives, ensure the "Extract Member" field exactly matches the filename inside the archive.
  • Permissions: If a binary is present but not working, the system will attempt to set its permissions to 0o750. Ensure Odoo has write access to its own data directory.
