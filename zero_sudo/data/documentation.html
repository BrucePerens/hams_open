Zero-Sudo Security: User Guide

The zero_sudo module is the foundational security layer of the platform. It enforces a "Zero-Sudo" policy, ensuring that background tasks and automated processes operate with minimal, strictly-defined permissions.

Key Security Features

Service Accounts

A "Service Account" is a special user profile created for automated tasks (like sending emails or backing up data). These accounts are hardened to prevent misuse:

  • Web Login Block: Service Accounts are strictly forbidden from logging in through a web browser. Any attempt to do so is instantly blocked and logged for auditing.
  • Automated Hardening: Every Service Account is assigned a complex, randomized password that is impossible to guess, ensuring only the intended automated system can use it.
  • How to Identify: You can manage these accounts in Settings > Users & Companies > Users by toggling the "Is Service Account" checkbox.

Safe Configuration & Parameter Whitelisting

To prevent sensitive data leaks, the system uses a strict whitelist for configuration parameters. Parameters like encryption keys or master passwords cannot be accessed or modified via standard tools unless they are explicitly registered in the security core. This protects against several classes of system-level attacks.

Centralized Security Logs

The system maintains a centralized audit trail for critical security events. Administrators can monitor these in Settings > Security Logs. Tracked events include:

  • Blocked Login Attempts: When a Service Account attempts to access the web UI.
  • Privilege Escalation Violations: When an automated process attempts to gain full system administrator rights.
  • Restricted Parameter Access: When an unauthorized attempt is made to read or write sensitive system configurations.
  • Cache Invalidation: Records system-wide data refreshes.

Safe Cache Invalidation

To optimize performance without compromising security, the system provides a granular way to refresh data caches. Users with appropriate permissions can safely trigger a refresh for specific parts of the system, ensuring that data is always up-to-date without needing full administrative rights. This prevents "stale data" issues while maintaining strict isolation between different parts of the platform.

Noisy Table Management

System administrators can flag specific technical tables as "Noisy" to streamline security audits. This reduces false positives by indicating that certain tables are known to have high-frequency, non-sensitive data changes.

Frequently Asked Questions

"I get an 'Access Denied' error when logging in!"

If you see an error referring to access_denied_service, you are trying to log in with a Service Account. These accounts are for background tasks only. Please use your standard employee account.
