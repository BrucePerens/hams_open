Zero-Sudo Security Core: User Guide [@ANCHOR: doc_title]

The zero_sudo module is the foundational security layer for our entire system. It acts as a "security cop," ensuring that every part of the software only has the permissions it absolutely needs to do its job. It implements two major security rules: Zero-Sudo Architecture and Service Account Isolation.

What is "Zero-Sudo"?

In most software, developers can easily "become the administrator" to perform complex tasks. This is dangerous because if one part of the system is hacked, the attacker can use this "sudo" power to take over everything. Our system forbids this. Instead, we use "Service Accounts" — special accounts that have very limited, specific permissions. This means that even if one component is compromised, the damage is strictly contained.

Key Features and Functionality

1. Service Accounts: Background Workers

Service accounts are special user records designed for background tasks and external systems (like backup tools, helpdesk bots, or automated reporters). Unlike human users, service accounts are highly specialized.

  • Identification: As an administrator, you can identify these accounts by looking for the "Is Service Account" checkbox on the User form under the Settings > Users & Companies > Users menu.
  • Web Isolation: For security, service accounts are physically blocked from logging into the website interface. If someone tries to use a service account's credentials in a web browser, the system will immediately terminate the session and redirect them to the login page with an error message.
  • External Access: While blocked from the web interface, service accounts can still communicate with the system via secure API channels (like JSON-RPC) to perform their assigned duties.

2. Safe Configuration Whitelisting

To prevent sensitive data leaks, this module enforces a "whitelist" for system configuration settings. In many systems, an attacker might try to trick the software into revealing secrets like database passwords or encryption keys. Zero-Sudo prevents this by only allowing access to settings that developers have explicitly marked as "safe" and "non-sensitive."

3. Centralized Communication Channels

The module provides specialized "bridge" accounts for common tasks that require slightly more permission, such as sending emails or interacting with the internal messaging system (the "Chatter"). By using these centralized bridges (like the Central Mail Service Account), other parts of the system don't need to be given broad administrator powers just to send a notification.

4. Automatic Documentation Library

The module also powers the very documentation system you are using right now! It automatically scans every installed module for manuals and guides, and then "bootstraps" them into this library. This ensures that your documentation is always up-to-date and centrally located, without requiring manual effort during installation.

5. Multi-Website Security

Zero-Sudo is designed to be fully aware of environments where multiple websites are running on a single Odoo instance. The security restrictions it imposes, such as blocking service accounts from web logins, are applied globally. This means that if an account is flagged for background tasks, it cannot be used to log into any of the websites managed by the system, ensuring consistent security across all digital properties.

6. Security and Secrets

The module includes advanced protections for the system's most sensitive data. Cryptographic secrets and master keys are managed using high-performance, cached methods that bypass standard database tables. This prevents attackers from using techniques like "SQL injection" or template manipulation to steal the keys that protect your data. These protections ensure that your most critical security information remains safe even if other parts of the system are under attack.

Operating the Module

Managing Service Accounts

  1. Navigate to Settings > Users & Companies > Users.
  2. Open a user record.
  3. In the Extra Rights section (or similar, depending on your view), you will see the Is Service Account checkbox.
  4. Check this box for accounts used by automated scripts or external integrations.
  5. Uncheck this box for accounts used by real people who need to log in via a browser.

Note: Converting a human user to a service account will immediately prevent them from logging in via the web.

Common Troubleshooting

  • Error: "Access Denied" (access_denied_service): If you see this error after trying to log in, it means the account you are using is flagged as a "Service Account." Service accounts are for background tasks only and cannot use the web interface. You must use a human user account to access the web interface.
  • Missing Manuals: If documentation for a specific module is missing, ensure the module is fully installed and that it correctly utilizes the knowledge_docs manifest property in its configuration file. The documentation is updated automatically whenever the system starts or a module is upgraded.
  • Parameter Access Errors: If an automated process fails because it cannot read a "System Parameter," it is likely because that parameter is not on the approved "safe list" or it contains sensitive keywords (like "password" or "secret") that are blocked for security reasons.

Technical Appendix

Developers looking to integrate with the Zero-Sudo core should consult the README.md in the zero_sudo module folder for details on the Python API (_get_service_uid, _get_system_param, etc.) and the Service Account provisioning patterns.
