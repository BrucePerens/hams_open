1. Overview

The Caching module implements a global, root-scoped Service Worker (/sw.js) that proxies and caches frontend assets locally in the browser to provide near-instant load times.

2. Integration Rules

  • Automatic Caching: Assets placed in your module's static/ directory are cached automatically. This includes JavaScript, CSS, and images.
  • No Competing Workers: DO NOT attempt to register another Service Worker. The browser only allows one worker per scope.
  • WebSockets: ws:// and wss:// protocols are hardcoded to bypass the proxy to ensure real-time communication is not interrupted.
  • Dynamic Large File Prohibition: The worker mathematically calculates an active quota limit (default 35MB). Heavy media MUST route via /web/image or /web/content as these routes are explicitly bypassed.
  • Layout Injection: The service worker registration script is injected globally into the frontend website.layout.
  • Zero-Sudo Architecture: The module uses micro-privilege service accounts (caching.user_caching_service) for administrative tasks like system parameter access and filesystem scanning, adhering to strict security protocols.

3. Configuration

Administrators can configure the caching behavior through the Website Settings:

  • Safe Quota (MB): Configure the browser storage limit for the service worker.
  • Force Cache Invalidation: Increment the version to force users' browsers to immediately wipe their cache.

4. Technical Details

The Service Worker intercepts requests matching /web/assets/ and /*/static/. It uses a Cache-First strategy for these assets, falling back to the network and then caching the result if it's within the size limit.

Cache invalidation is handled by an mtime-based versioning scheme. The /sw.js route dynamically injects the latest filesystem modification time from the server's static/ directories, forcing the browser to update the Service Worker whenever assets change.
