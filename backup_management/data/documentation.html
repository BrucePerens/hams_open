Unified Backup Management Manual

This module implements the Best-in-Class Hybrid Architecture for system and database backups. It orchestrates Kopia for file/system backups and pgBackRest for PostgreSQL continuous WAL archiving.

1. Self-Healing Deployment

Odoo expects the kopia and pgbackrest packages to be installed at the host OS layer. If an administrator fails to install Kopia during provisioning, the application will automatically perform a Just-In-Time (JIT) download of the static Kopia binary from GitHub, placing it safely into the Odoo datastore to guarantee operation. Documentation is automatically bootstrapped during module registration via the knowledge_docs facility. [@ANCHOR: backup_doc_injection]

2. Configurations & Orchestration

Navigate to Backups > Configurations. You can define targets, storage backend credentials (S3/B2), and retention policies for both Kopia and pgBackRest. For Kopia, the Target / Stanza field should be the absolute path to your repository. For pgBackRest, it should be the name of the stanza.

If using Kopia, enter your repository password in the configuration; it will be symmetrically encrypted at rest. This configuration process is detailed in the Backup Configuration Journey. [@ANCHOR: journey_backup_config_sync]

You can actively trigger backups, apply retention policies (Kopia only), or manually synchronize snapshot metadata using the action buttons in the configuration header. These operations are executed asynchronously by a background worker to ensure system responsiveness.

3. Synchronization

The system automatically keeps your backup metadata up-to-date. An automated hourly cron job interrogates the underlying CLI tools (kopia snapshot list and pgbackrest info) to fetch the latest backup metadata and populate the Snapshots table. This ensures the Odoo dashboard always reflects the actual state of your storage backends. [@ANCHOR: UX_BACKUP_SYNC]

4. Restoration Assistance

Because restoration can be destructive, the system provides a controlled Restore Wizard. Navigate to any Snapshot record to trigger a restoration to a specific target path. [@ANCHOR: backup_trigger_restore]

See the Manual Restore Journey for more details. [@ANCHOR: journey_manual_restore_command]

Alternatively, navigate to any Snapshot record to view and copy the exact generated restoration command for that specific point in time.

5. Pager Duty Synergy & Stale Backups

This module features native synergy with the Pager Duty SRE suite. If an underlying backup engine fails to execute, or if the system detects that a configuration hasn't produced a new successful snapshot in over 26 hours, it will automatically open a Critical incident, ensuring the on-call administrator is immediately paged. This failure reporting logic is a core part of our SRE Stories. [@ANCHOR: story_failure_reporting]

6. Snapshot Size Anomaly Detection

Administrators can configure a Minimum Expected Size (MB) on each backup configuration. When a new snapshot is ingested, the system compares its physical byte size against this threshold. If the snapshot is anomalously small (indicating a missing mount or failed filesystem read), the system immediately fires an SRE alert to prevent a false sense of security.

7. Security Hardening

The module implements strict path validation. Backup targets and restore paths are checked against a forbidden list of system directories (like /etc, /root, etc.) to prevent accidental overwriting of critical system files. Furthermore, shell metacharacters and directory traversal patterns (..) are strictly prohibited in any path configuration to prevent command injection and unauthorized file access. All sensitive credentials, such as repository passwords and S3 secret keys, are encrypted at rest using industry-standard Fernet encryption.

All background operations are performed using a dedicated service account, adhering to the Zero-Sudo architecture. This ensures that the system operates with the minimum necessary privileges and maintains a robust audit trail.
