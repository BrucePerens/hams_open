1. Architecture Overview (CQRS)

The Pager Duty module operates on a Command Query Responsibility Segregation (CQRS) model. Odoo acts as the control plane to manage configurations, while an isolated, standalone Python daemon (generalized_monitor.py) executes the actual polling and health checks outside of Odoo's web workers. [@ANCHOR: doc_inject_pager_duty]

2. Configuration Wizard

Administrators can configure all monitoring endpoints graphically. Navigate to Admin > System Health > Monitoring Checks. Click Edit YAML Config File to synchronize the database records with the daemon's physical pager_config.json file. [@ANCHOR: generalized_pager_config] [@ANCHOR: test_pager_view]

3. Supported Check Types

  • System Resource: Monitors Disk, Memory, CPU, IO Wait, and Hypervisor Steal percent limits.
  • DNS Resolution: Performs un-cached, full-chain root server lookups (via dig +trace).
  • HTTP / HTTP/3 (QUIC): Validates web server responsiveness across modern and legacy web protocols.
  • TCP & UDP Port Monitoring: Asserts raw datagram and stream connectivity to any custom port, supporting hexadecimal payload injection.
  • Native Protocol Handshakes: Built-in support for executing strict protocol validations against LDAP, NTP, Redis, RabbitMQ (AMQP), XML-RPC, and JSON-RPC.
  • PostgreSQL & MySQL/MariaDB: Validates database connection pools directly via psycopg2 and pymysql by logging in and executing non-destructive SELECT 1; queries.
  • SSL/TLS Certificate Expiry: Alerts administrators if the domain's certificate expires within a critical threshold.
  • Anomaly Detection: Evaluates a custom SQL query. If the integer drops below the threshold, it fires an alert.
  • Synthetic Journeys: Executes a custom shell script or headless browser test, alerting on non-zero exit codes.
  • Pre-Flight & Dry-Runs: Safely simulates critical infrastructure maintenance including certbot --dry-run, pg_dump -s, nginx -t, and outbound SMTP authentications.
  • ICMP Ping: Basic network reachability testing for remote gateways and routers.
  • Docker Container Health: Monitors the native execution state of background Docker containers.
  • Systemd Services: Validates that critical host OS services remain active. Using the wildcard (*) target tracks all services globally and alerts on any that enter a failed state, with support for specific service exclusions.
  • SMART Disk Health: Integrates with smartctl via an airgapped JSON spool to predict impending SSD/HDD drive failures securely.
  • Playwright Synthetic Journeys: Executes headless Python Playwright scripts via an airgapped spooler daemon.
  • Sandboxed Bash & Executables: Executes arbitrary scripts and downloaded binaries inside a strict Bubblewrap (bwrap) sandbox, physically severing network and file system access unless explicitly toggled. Supports cryptographic SHA-256 verification of downloaded payloads before execution.
  • Memcached Server: Raw TCP socket handshakes to ensure the cache is responding to statistic inquiries.
  • SSH Handshake: Validates raw port accessibility and protocol compliance for remote access.

4. Autodiscovery Wizard

When deployed on a new node, the module features a one-click Autodiscovery engine. It actively scans the host operating system's process tree (Systemd) and physical mount points to automatically provision optimal baseline monitoring for PostgreSQL, Redis, RabbitMQ, Docker, local HTTP routing, and Disk Space usage.

5. Intelligent Calendar Routing

The module natively ties into the Odoo Calendar. By checking the "Is Pager Duty Shift" box on an event, the system automatically knows exactly who is on-call at any given minute. It fully supports complex recurrence rules (like "the first Tuesday of the month") and manages shift-handoffs seamlessly without requiring manual configuration files. [@ANCHOR: test_pager_notification]

6. Fail-Safes & Self-Healing

Checks can be configured with a Startup Grace Period (e.g., 120 seconds) to prevent false-positive alerts while Odoo is initially booting. If Odoo crashes entirely, the daemon utilizes an Airgapped SMTP Fallback and Discord/Slack Webhooks to bypass the ERP and alert the administrator directly.

Furthermore, the daemon features a self-healing dependency scanner. For example, if it is instructed to test a Zero Trust tunnel but the cloudflared binary is missing from the system, it will automatically download the static binary from GitHub and continue execution without requiring manual intervention. [@ANCHOR: synthetic_i18n]

7. Log Analysis & SRE Forensics

The module includes a chrooted Log Analyzer daemon that tails system logs in real-time. Administrators can define regex patterns to detect anomalies like filesystem corruption or kernel panics. The system also features a Splunk-like interactive search UI, allowing for rapid forensics across distributed log files directly from the Odoo backend. [@ANCHOR: test_log_analyzer_views] [@ANCHOR: test_log_api_i18n]

8. Advanced SRE Capabilities

  • Cascading Failure Suppression: Link checks together via "Parent Check" dependencies. If the master database goes offline, downstream HTTP checks will be automatically silenced to prevent alarm fatigue. [@ANCHOR: auto_resolve_incidents]
  • Maintenance Windows: Schedule explicit silences via Maintenance Start/End dates to mute alerts during planned infrastructure work.
  • Push Monitoring (Heartbeat): Use the /api/v1/pager/heartbeat/<uuid> endpoint to actively listen for successful "pings" from external backup scripts. If a ping is missed, the daemon raises an incident. [@ANCHOR: report_incident_rate_limit]
  • Multi-Tier Escalation: A background cron job constantly evaluates open incidents. If a critical incident remains unacknowledged for over 15 minutes, it automatically escalates the alert directly to the entire administrative group, bypassing the scheduled shift. [@ANCHOR: test_pager_escalation]
  • Analytics: Incident records automatically track Mean Time To Acknowledge (MTTA) and Mean Time To Resolve (MTTR) metrics for operational review. [@ANCHOR: action_acknowledge_incident] [@ANCHOR: pager_board_data] [@ANCHOR: test_pager_board_url]

9. Helpdesk Integration

The Pager Duty module can automatically dispatch tickets to internal Helpdesk systems. This is managed via an Adapter pattern that dynamically resolves the target model (e.g., hams_helpdesk.ticket) and creates a unified tracking record. If the target system is unreachable, it automatically triggers a Direct SMTP Fallback to ensure the alert is never lost. [@ANCHOR: pd_helpdesk_adapter]
