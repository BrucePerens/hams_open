Hams Testing Infrastructure

This module provides the core testing facilities for the repository, ensuring that our software is reliable, secure, and functions as expected in a production-like environment.

Real Transaction Testing

Most tests in Odoo are "pretend" tests—they don't actually save any data to the database permanently. While this is fast, it can sometimes hide bugs that only appear when data is truly saved.

Our Real Transaction Testing Facility allows developers to write tests that behave exactly like the real world. It saves data for real during the test, then carefully cleans it all up when the test is finished.

Database Leak Protection

Because these tests save real data, we have to be very careful not to leave any "trash" behind in the database. The system automatically tracks every record created during a test and deletes it afterward. As an extra safety measure, it counts all the rows in every table before and after the test. If any extra rows are left over, the test fails, alerting the developer that something wasn't cleaned up properly.

Noisy Tables

Some tables in Odoo are naturally "noisy"—they get new data all the time from background processes (like system logs or internal message tracking). If the leak protector caught these, tests would fail for no reason.

Administrators can manage which tables are ignored by the leak protector using the Noisy Tables interface ([@ANCHOR: UX_NOISY_TABLE_MANAGEMENT]). You can find this under Settings > Technical > Database Structure > Noisy Tables.

  • Table Name: The internal name of the database table to ignore (e.g., ir_logging).
  • Active: If unchecked, the table will no longer be ignored, and the leak protector will start checking it again.
  • This management interface is crucial for ensuring tests pass without false positives ([@ANCHOR: test_noisy_table_tour]).

External Service Testing

Many of our modules talk to external services or run background processes (daemons). This module provides the tools to automatically start these services before a test begins and shut them down afterward. It also makes sure the service is actually "awake" and responding before the test starts sending it commands.

User Interface (UI) Tours

To make sure our screens and buttons work correctly for users, we use UI Tours. These are automated scripts that "drive" the web browser, clicking buttons and typing into forms just like a human would. This ensures that even the most complex parts of the interface remain easy to use and bug-free.

Diagnostic Tools

If a UI tour fails, the system provides a specialized "Skeleton Dump" in the error logs. This dump strips away the visual clutter and provides a simplified view of the interactive elements on the page at the time of failure. This helps developers quickly identify if a button was missing or if the page redirected unexpectedly.
